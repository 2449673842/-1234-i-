### [009] 字体中心/属性字体修改无效，误差棒线宽难以统一调整

* **记录时间**: `2026-06-26`
* **涉及项目**: `4方差感知图` / 项目多图模式 / 右侧栏编辑
* **具体表现**:
  ```text
  字体中心和属性编辑中修改字体大小/字体后没有明显反应，刷新或重渲染后容易回退；
  误差棒这类线段没有集中入口统一调整线宽。
  ```
* **原因分析**:
  * [x] **刻度文字 gid 不稳定**: `xtick.* / ytick.*` 是 Matplotlib 每次 draw 后生成的 Text artist，直接修改这些临时文本对象不如修改稳定的 `axis.x.* / axis.y.*`。
  * [x] **项目模式 local patch revision 冲突**: 前端乐观应用 local patch 时提前把本地 `revision + 1`，同一请求再带旧 `baseRevision` 发后端，容易触发冲突或状态回退。
  * [x] **误差棒类型覆盖不足**: Matplotlib 的误差棒线段常落在 `LineCollection`，旧 manifest 没读取/暴露 `collection.linewidth`，组件级线宽无法稳定编辑。
  * [x] **分组编辑过度依赖脚本语义**: 没有 label/group 的误差棒不会进入逻辑分组，用户找不到统一编辑入口。
* **解决方案**:
  1. 修改 `src/components/RightSidebar.tsx`：字体中心的 X/Y 刻度组优先使用稳定的 `axis.x.* / axis.y.*`，并把字号/字体/颜色转换为 `tick_labelsize / tick_labelfamily / tick_labelcolor`。
  2. 修改 `src/components/RightSidebar.tsx`：字体预设也复用同一套稳定 patch 生成逻辑，不再逐个写临时 tick text。
  3. 修改 `src/App.tsx`：项目模式 local patch 乐观更新 SVG/manifest/editLog 时不提前增加 revision，后端响应才是 revision 来源。
  4. 修改 `renderer/introspector.py`：`collection` 读取并暴露 `linewidth`，用于误差棒/集合线宽编辑。
  5. 修改 `src/components/RightSidebar.tsx`：将旧“分组编辑”入口替换为“组件中心”，按真实图元聚合线条、集合/误差棒、柱形/面、边框/网格。
* **验证方式**:
  ```text
  最小重放测试：
  axis.x.0 tick_labelsize -> 18.0
  axis.x.0 tick_labelfamily -> Arial
  collection.0.0 linewidth -> 4.0
  warnings -> None

  C:\Users\SZC\.conda\envs\Machine-learning\python.exe -m py_compile renderer\introspector.py
  npm run lint
  npm run build
  ```
* **当前状态**: `已解决；需要重启 npm dev 服务后加载最新 server/frontend bundle`

### [010] 项目编辑全局失效：所有改图操作看起来没有反应

* **记录时间**: `2026-06-26`
* **涉及项目**: `4方差感知图` / 项目多图模式 / `/api/figure/patch`
* **具体表现**:
  ```text
  字体、颜色、线宽、画布尺寸等操作都没有明显反应；
  前端没有清楚提示错误，用户看到的是“改啥都没反应”。
  ```
* **原因分析**:
  * [x] **项目 Figure revision 与 Session revision 分叉**:
    1. `sessions.revision` 已经随 patch 增加到 11。
    2. `project_figures.revision` 仍停留在 1。
    3. 项目打开后的自动重渲染路径又把前端 `projectFigures[fig].revision` 写死为 1。
    4. 后续 patch 携带 `baseRevision=1`，后端看到 session revision=11 后返回 `409 Revision conflict`。
  * [x] **前端错误处理不足**: `handlePatch()` 对非 success 响应没有明确日志，导致冲突表现为“没反应”。
  * [x] **单机工具不适合硬性 revision 拒绝**: 当前产品没有多人协作，严格冲突阻断收益低，但会直接破坏核心编辑闭环。
* **解决方案**:
  1. 修改 `server.ts`：`/api/figure/patch` 不再因 stale `baseRevision` 硬拒绝；改为应用到最新 session，并返回 `revision_mismatch` warning。
  2. 修改 `server.ts`：patch 成功后同步更新 `project_figures.revision`。
  3. 修改 `server.ts`：项目重渲染返回每张 Figure 的 `revision` 和 `editLog`。
  4. 修改 `db.ts`：`replaceProjectFiguresAndSessions()` 写入 `project_figures.revision`。
  5. 修改 `src/App.tsx`：项目加载、自动重渲染、手动重渲染、撤销/重做重放均优先使用服务端返回的 revision，不再写死 `1` 或盲目 `+1`。
  6. 修改 `src/App.tsx`：非 success patch 响应写入日志；若仍遇到 conflict，会同步本地 revision 并提示重新应用。
* **验证方式**:
  ```text
  当前运行中的 localhost:3000 API probe：
  render 200 success <sessionId> 1
  patch  200 success revision=2
  warnings=[revision_mismatch expectedRevision=1 receivedRevision=0]
  title fontsize=18

  npm run lint
  npm run build
  ```
* **当前状态**: `已解决；当前 3000 dev server 已重启并验证`

  ```text

  > [错误] 脚本执行失败: name '__file__' is not defined
  ```
* **原因分析**:
  * [x] **平台环境问题**: 平台底层的 Python 执行环境在初始运行时缺失 `seaborn` 依赖模块。

    2. **硬编码本地绝对路径**: 脚本中强行硬编码了用户本地电脑的绝对路径 `C:\Users\SZC\OneDrive\Desktop\课题相关\...`。在平台运行环境上这些路径根本不存在。
* **解决方案**:
  1. **环境补全**: 在平台 Python 环境中安装 `seaborn`（或者在 `requirements.txt` 中配置并确保其正确安装生效）。
  2. **重构脚本文件定位逻辑**: 
     - 彻底移除对 `__file__` 以及本地绝对路径的依赖。
     - 改为使用平台注入的 `_uploaded_file_paths` 字典来动态获取已上传的文件路径。
     - 例如，将：
       ```python
       time_df = pd.read_csv(TIME_DATA)
       fl9_df = pd.read_csv(FL9_DATA)
       ```
       重构为使用平台上传好的对应文件：
       ```python
       time_df = pd.read_csv(_uploaded_file_paths["时间轨迹_样点分类数据.csv"])
       fl9_df = pd.read_csv(_uploaded_file_paths["FL9_classification_modeling_matrix.csv"])
       ```
       *(注：需要确保相关数据文件已上传到该项目中。)*
* **当前状态**: `已解决`（已定位成因并提供脚本重构方案建议）

### [007] 配色中心修改颜色后没有应用到图上

* **记录时间**: `2026-06-25`
* **涉及项目**: `10聚类` / 项目多图模式 / 配色中心
* **具体表现**:
  ```text
  在配色中心修改颜色后，右侧控件看似触发了修改，但对应 Figure 上的图元颜色没有变化；
  部分图刷新或重新渲染后颜色还会回退。
  ```
* **原因分析**:
  * [x] **语义扫描覆盖不足**: 旧扫描器只识别 `SEMANTIC_COLORS`，真实脚本里的 `CLUSTER_COLORS`、`GROUP_COLORS` 等通用颜色字典不会生成 palette。
  * [x] **绑定器依赖显式 group**: 脚本使用 `df["cluster"].map(CLUSTER_COLORS)` 这类向量化配色时，没有 `ax.scatter(... label=...)` 可供 AST 建 group，导致 `manifest.bindings=[]`，配色中心没有实际 gid 目标。
  * [x] **SVG 本地 patch 被 inline style 覆盖**: matplotlib SVG 常把 `fill/stroke/font-size` 写在 `style` 中，只改 attribute 不一定生效。
  * [x] **项目脚本未持久化**: code_patch 只更新 figure session 的 script，未同步 `projects.script`，刷新或多图重建后会回退到旧颜色。
* **解决方案**:
  1. 修改 `renderer/semantic_scanner.py`：扫描任意包含 `#RRGGBB` 的运行时/静态颜色字典，并生成 `dict_<key>` palette。

  1. 修改 `renderer/introspector.py`：不再把空标题 `title.0` 暴露为可编辑对象，只绑定有文本内容的标题。
  2. 修改 `renderer/introspector.py`：字体、标题、轴标签、tick label 等布局相关 editLog 重放后自动执行 `fig.tight_layout()`。
  3. 清理当前 `10聚类 / fig_4` 数据库 editLog 中的坏记录：
     ```text
     title.0 visible=false
     axis.y.0 visible=false
     ```
