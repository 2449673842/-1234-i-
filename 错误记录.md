### [001] "10.聚类图" 渲染报错 (No module named 'seaborn' & NameError: name '__file__' is not defined)

* **记录时间**: `2026-06-25 11:27:00`
* **上传的数据文件**:
  1. `Table_Hclust_time_specific_spearman.csv` (及其它 9 个相关的关联 CSV 文件)
* **使用的 AI 转义/绘图脚本**: 诊断报告中的 "AI 转义后 / 当前平台脚本"
* **具体报错信息**:
  ```text
  > [渲染错误] 脚本执行失败: No module named 'seaborn'
  > [错误] 脚本执行失败: name '__file__' is not defined
  ```
* **原因分析**:
  * [x] **平台环境问题**: 平台底层的 Python 执行环境在初始运行时缺失 `seaborn` 依赖模块。
  * [x] **转义脚本编写问题**:
    1. **使用未定义变量**: 脚本中使用了 `Path(__file__)`。由于平台在后端渲染时采用 `exec(script, ns, ns)` 动态执行脚本内容，并不存在物理文件上下文，因此全局命名空间内没有 `__file__` 变量，导致抛出 `NameError: name '__file__' is not defined`。
    2. **硬编码本地绝对路径**: 脚本中强行硬编码了用户本地电脑的绝对路径 `C:\Users\SZC\OneDrive\Desktop\课题相关\...`。在平台运行环境上这些路径根本不存在。
* **解决方案**:
  1. **环境补全**: 在平台 Python 环境中安装 `seaborn`（或者在 `requirements.txt` 中配置并确保其正确安装生效）。
  2. **重构脚本文件定位逻辑**: 
     - 彻底移除对 `__file__` 以及本地绝对路径的依赖。
     - 改为使用平台注入的 `_uploaded_file_paths` 字典来动态获取已上传的文件路径。
     - 例如，将：
       ```python
       time_df = pd.read_csv(TIME_DATA)
       fl9_df = pd.read_csv(FL9_DATA)
       ```
       重构为使用平台上传好的对应文件：
       ```python
       time_df = pd.read_csv(_uploaded_file_paths["时间轨迹_样点分类数据.csv"])
       fl9_df = pd.read_csv(_uploaded_file_paths["FL9_classification_modeling_matrix.csv"])
       ```
       *(注：需要确保相关数据文件已上传到该项目中。)*
* **当前状态**: `已解决`（已定位成因并提供脚本重构方案建议）

### [002] 字体中心 / 多选批量修改刻度字体后刷新失效

* **记录时间**: `2026-06-25`
* **涉及项目**: `10聚类`
* **具体表现**:
  ```text
  选中 Y 轴刻度文字或在字体中心修改刻度字体大小后，界面短暂变化；
  刷新或重新渲染后字体大小恢复原状。
  ```
* **原因分析**:
  * [x] **平台前端问题**: `RightSidebar.tsx` 的 `renderNumberInput` 虽然接收了专门的 `onValue` 回调，但提交时调用的是全局 `commitNumberDraft -> handlePatch`，导致字体中心把 `font-center-yticks`、`font-center-xticks`、`batch` 这类 UI 控件 ID 写入 editLog。
  * [x] **平台后端兼容问题**: 后端只能重放真实 matplotlib gid，例如 `axis.y.0 / tick_labelsize`，无法识别历史日志中的虚拟 UI id。
* **解决方案**:
  1. 修改 `src/components/RightSidebar.tsx`：数字输入提交时调用传入的 `onValue`，确保字体中心和多选批量编辑会展开为真实 patch。
  2. 修改 `renderer/introspector.py`：增加旧日志兼容层，将 `font-center-yticks / font-center-xticks` 自动展开到 `axis.y.* / axis.x.*` 的 `tick_labelsize / tick_labelfamily / tick_labelcolor`。
* **验证方式**:
  ```text
  使用 10聚类 项目的真实脚本、真实上传文件和 Figure 3 editLog 重放；
  确认 Y 轴 tick 字号能按 editLog 统一恢复。
  npm run lint
  npm run build
  py_compile renderer/introspector.py
  ```
* **当前状态**: `已解决`

### [003] 改图时无进度反馈，用户误以为卡死

* **记录时间**: `2026-06-25`
* **涉及页面**: 编辑器 / 画布预览 / 项目多图模式
* **具体表现**:
  ```text
  修改字体、坐标轴、图例等 backend_patch 参数后，Python 重放需要数秒；
  页面缺少明确进度提示，用户无法判断是在运行还是卡住。
  ```
* **原因分析**:
  * [x] **平台交互问题**: 项目模式下 `handlePatch` 没有设置 `projectIsRendering`，也没有写入渲染日志。
  * [x] **状态展示不足**: 工作区只在少数手动渲染路径显示 `Python 引擎运行中...`，属性修改重渲染没有统一进度层。
* **解决方案**:
  1. 修改 `src/App.tsx`：backend patch、code patch、项目恢复渲染、项目整图渲染均设置 `renderProgressText` 和日志。
  2. 修改 `src/components/MainWorkspace.tsx`：顶部增加运行状态、耗时；画布中增加浮层进度提示。
  3. 修改 `src/index.css`：增加 `render-progress` 进度条动画。
* **验证方式**:
  ```text
  npm run lint
  npm run build
  ```
* **当前状态**: `已解决`

### [004] 项目多图模式撤销 / 重做按钮不作用于当前 Figure

* **记录时间**: `2026-06-25`
* **涉及项目**: `10聚类`
* **具体表现**:
  ```text
  在 Figure 4 修改图形后，点击顶部“撤销”按钮没有恢复当前图；
  样式突变后无法通过按钮回退。
  ```
* **原因分析**:
  * [x] **平台状态模型问题**: 顶部撤销/重做按钮只接了旧的单图 `useFigureSession` 历史和 spec 历史，没有优先调用当前项目 Figure 的 `projectHistory`。
  * [x] **状态更新问题**: `handleProjectUndo / handleProjectRedo` 里曾直接修改 history 数组对象，容易造成 React 状态不同步。
  * [x] **刷新后历史丢失问题**: 浏览器刷新后内存中的 undo 栈会丢失，但服务端 editLog 仍存在。
* **解决方案**:
  1. 修改 `src/App.tsx`：项目模式下撤销/重做优先作用于当前 `activeFigureId`。
  2. 将 `projectHistory` 更新改为不可变更新。
  3. 增加兜底撤销：如果内存历史栈为空但当前 Figure 有 editLog，则删除最后一组相同 timestamp 的操作并重放。
  4. 撤销/重做时显示进度并重建项目 SVG。
* **验证方式**:
  ```text
  npm run lint
  npm run build
  ```
* **当前状态**: `已解决`

### [005] Figure 4 标题 / Y 轴标签跑到画布外或消失

* **记录时间**: `2026-06-25`
* **涉及项目**: `10聚类`
* **具体表现**:
  ```text
  Figure 4 修改后，标题跑到画布外；
  左侧 ylabel.0 / PCoA2 标签不正常，甚至从 SVG 中消失。
  ```
* **原因分析**:
  * [x] **错误 editLog 污染**:
    1. `global / figure.width_in = 6`、`global / figure.height_in = 6` 改变了整张 Figure 的画布比例。
    2. `title.0 visible=false` 隐藏的是 matplotlib 的空中心标题，不是实际左标题 `title.left.0`，会触发布局异常。
    3. `axis.y.0 visible=false` 会隐藏整个 YAxis，包括 `ylabel.0`，所以 `PCoA2` 从 SVG 中消失。
  * [x] **平台内省问题**: 空标题对象也被绑定为可编辑对象，用户容易误选。
  * [x] **布局重放问题**: 脚本执行过 `fig.tight_layout()` 后，平台再重放字体大小修改，但没有重新执行布局，导致放大字号后标签越界。
* **解决方案**:
  1. 修改 `renderer/introspector.py`：不再把空标题 `title.0` 暴露为可编辑对象，只绑定有文本内容的标题。
  2. 修改 `renderer/introspector.py`：字体、标题、轴标签、tick label 等布局相关 editLog 重放后自动执行 `fig.tight_layout()`。
  3. 清理当前 `10聚类 / fig_4` 数据库 editLog 中的坏记录：
     ```text
     title.0 visible=false
     axis.y.0 visible=false
     global figure.width_in / figure.height_in = 6
     ```
  4. 调用项目重建接口重新生成 Figure 4 SVG/manifest。
* **验证方式**:
  ```text
  直接重放 10聚类 Figure 4：
  hasTitle = true
  hasYLabel = true
  badTitle = 0
  badAxisY = 0
  badSize = 0

  npm run lint
  npm run build
  py_compile renderer/introspector.py
  ```
* **当前状态**: `已解决`

### [006] 刷新后旧的坏 editLog 从 sessionStorage 复活

* **记录时间**: `2026-06-25`
* **涉及项目**: 项目多图模式
* **具体表现**:
  ```text
  数据库中已经清理了异常 editLog，但浏览器刷新后仍可能用 sessionStorage 中的旧 projectFigures 自动重建，
  导致已删除的坏状态重新出现。
  ```
* **原因分析**:
  * [x] **平台恢复逻辑问题**: 自动恢复项目预览时优先使用浏览器 sessionStorage 中的 `projectFigures.editLog`，而不是服务端数据库的最新 editLog。
* **解决方案**:
  1. 修改 `src/App.tsx`：自动恢复项目预览时先请求 `/api/projects/:id`，以服务端项目 figures/editLog 为准。
  2. 只有服务端没有返回 figures 时，才回退使用本地 `projectFigures`。
  3. sessionStorage 继续剥离大型 SVG，只保存轻量状态，避免刷新卡顿。
* **验证方式**:
  ```text
  调用 /api/projects/58dd5b9f-6cb2-4be5-84f6-8f31265eb015：
  fig4EditCount = 29
  badGlobals = 0

  npm run lint
  npm run build
  ```
* **当前状态**: `已解决`
